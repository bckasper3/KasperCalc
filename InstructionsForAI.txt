The core problem is CoolProp throws C++ exceptions that bubble up as Python exceptions, and they're not always predictable — a point that works fine at one pressure might crash at a nearby pressure for no obvious reason. Recommendation:
For each (T, P) point:
  - Wrap in try/except
  - On exception, store NaN for all Float32 properties
  - Store a specific phase flag value to indicate "failed" (distinct from "two-phase")
  - Log the failed point (fluid, T, P, error message) to a separate error log file
  - Continue — never let a single point crash the run
Keep your error log per-fluid so after generation you can inspect it and decide if the failures are acceptable or need a denser grid in that region. Near critical points expect a lot of failures — that's normal.
For the overall run, checkpoint after every fluid completes, not after every point. Write a simple completed.json that lists finished fluids so a restart skips them.

Two-Phase Region Strategy
This is the more interesting decision. Your options are:
Option A — Store NaN, skip it
Simple. The frontend just shows "two-phase region" as a blank or hatched area. Clean, honest, easy to implement. Downside is users can't look up mixed-phase properties, which is sometimes useful.
Option B — Store saturated liquid and vapor separately
For every point that falls inside the two-phase dome, instead of one record you store two — one at Q=0 (saturated liquid) and one at Q=1 (saturated vapor). The frontend knows to display both endpoints. This is probably the most useful for engineers.
Option C — Store quality-weighted averages
Requires assuming a quality value which you don't have, so this is generally a bad idea for a lookup table. Dismiss this one.

Recommendation: Option B with NaN fallback
Concretely:
For each (T, P) point:
  1. Evaluate normally
  2. If CoolProp returns phase = "two-phase":
       - Re-evaluate at same T with Q=0 → store as saturated liquid record
       - Re-evaluate at same T with Q=1 → store as saturated vapor record
       - Store a phase flag = 2 in the main grid to indicate "see saturation data"
  3. If evaluation throws exception:
       - Store NaN, phase flag = 255 (failed)
       - Log it
This means your saturation curve data (which you're already computing in the fluid profile step) does double duty — it's both the boundary metadata AND the property source for two-phase grid points. You avoid storing redundant data.
